Normalization Notes
1NF All attributes are atomic (no multi-valued fields). Blood group stored as single enum value. Parameters stored in separate report_parameter table.
2NF All non-key attributes depend on the full primary key. No partial dependencies (each FK is a full table reference, not composite key partial dependency).
3NF No transitive dependencies. Doctor name not stored in test_order (only doctor_id FK). Test price stored only in lab_test, not duplicated in test_order.
BCNF Every determinant is a candidate key. license_number → doctor_id (unique constraint), phone → patient_id (unique constraint). Both are candidate keys.
Key SQL Queries with Explanations
Q1 · Critical Reports with Patient Emergency Contact
Purpose: Identify patients needing emergency notification
SELECT
    lr.report_id,
    CONCAT(p.first_name, ' ', p.last_name) AS patient_name,
    p.emergency_contact,
    lt.test_name,
    lr.result_summary,
    lr.report_date
FROM lab_report lr
JOIN test_order to_ ON lr.order_id    = to_.order_id
JOIN patient p      ON to_.patient_id = p.patient_id
JOIN lab_test lt    ON to_.test_id    = lt.test_id
WHERE lr.is_critical = TRUE
ORDER BY lr.report_date DESC;
→ Returns 2 rows: Troponin I (Mohit Joshi), ECG AF (Rahul Verma)
Q2 · Tests by Category with Abnormality Rate (GROUP BY + HAVING)
Purpose: Workload planning and quality monitoring
SELECT
    lt.category,
    COUNT(lr.report_id)                              AS total_reports,
    SUM(CASE WHEN lr.is_abnormal THEN 1 ELSE 0 END) AS abnormal_reports,
    ROUND(AVG(lt.price), 2)                          AS avg_test_price
FROM lab_report lr
JOIN test_order to_ ON lr.order_id = to_.order_id
JOIN lab_test lt    ON to_.test_id = lt.test_id
GROUP BY lt.category
ORDER BY total_reports DESC;
→ Biochemistry: 4 reports | Hematology: 3 reports | Radiology: 2 reports
Q3 · Patients with Critical Reports (JOIN vs Subquery comparison)
Purpose: Demonstrate two approaches to the same requirement
-- JOIN approach (better performance on large datasets):
SELECT DISTINCT p.first_name, p.last_name, p.emergency_contact
FROM patient p
JOIN test_order to_ ON p.patient_id = to_.patient_id
JOIN lab_report lr  ON to_.order_id = lr.order_id
WHERE lr.is_critical = TRUE;

-- Subquery approach (more readable for analysts):
SELECT CONCAT(first_name,' ',last_name), emergency_contact
FROM patient
WHERE patient_id IN (
    SELECT to_.patient_id FROM test_order to_
    JOIN lab_report lr ON to_.order_id = lr.order_id
    WHERE lr.is_critical = TRUE
);
→ JOIN preferred: query planner can use index on test_order.patient_id; subquery forces sequential scan
Q4 · Relational Algebra → SQL Translation
σ(is_critical=TRUE)(lab_report) ⋈ test_order ⋈ patient
-- σ = WHERE filter
-- ⋈ = JOIN
-- π = SELECT projection

SELECT p.patient_id, p.first_name, p.last_name,
       lr.report_id, lr.result_summary
FROM lab_report lr
JOIN test_order to_ ON lr.order_id    = to_.order_id
JOIN patient p      ON to_.patient_id = p.patient_id
WHERE lr.is_critical = TRUE;
→ Demonstrates direct mapping from RA operators to SQL clauses

Why Access Logging Matters
PRIVACY Every access to a lab report — view, download, print, edit — is logged with role, timestamp, and IP. No report can be accessed without a trace.
COMPLIANCE Access logs are immutable (no UPDATE/DELETE allowed on access_log table). This mirrors HIPAA-style audit requirements in medical systems.
TRIGGER The audit_log table captures before/after values for every UPDATE on lab_report (via trigger). This enables forensic comparison if data is tampered with.
POLICY Unreleased reports (is_released=FALSE) are invisible to patients at the view layer. The trigger prevents release before doctor verification.
